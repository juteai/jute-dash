# Security And Privacy

## Default Posture

Jute is local-first. The hub binds to loopback by default and requires explicit configuration before exposing APIs to the local network. Public internet exposure is not a v1 feature.

## Network Exposure

- Development defaults to `127.0.0.1`.
- LAN exposure requires a clear config setting.
- Remote access requires an explicit future design for authentication, TLS, pairing, and audit logging.
- CORS should be narrow in production and open only in development.

## Secrets

Secrets are never committed, displayed, or sent to widgets.

v1 secret handling:

- store references in config or SQLite, such as environment variable names;
- read raw secret values only inside the hub;
- never include raw secret values in public config endpoints;
- redact secret references in logs where possible.

Future secret handling:

- OS keyring;
- OAuth device flow;
- mTLS identities;
- encrypted backup/export support.

## Voice Privacy

Voice is local-first by default.

Rules:

- raw microphone audio stays local by default;
- wake-word detection and voice activity detection run locally before any cloud provider is called;
- cloud speech-to-text and text-to-speech providers are opt-in per household or device profile;
- A2A agents receive final transcripts and redacted dashboard context only;
- raw audio, pre-roll buffers, and partial transcripts are not sent to A2A agents;
- voice logs exclude raw audio and raw transcripts by default;
- ambient mode avoids showing transcripts unless the user has enabled visible conversation history.

## Voice Provider Security

Voice Provider Packs are untrusted by default unless they are built into Jute.

Rules:

- never load arbitrary provider code into the hub process;
- do not use Go dynamic plugins for v1 voice providers;
- validate `jute.voice.provider.json` before showing a provider in settings;
- store provider secrets as references, not raw manifest values;
- reject provider endpoints that embed usernames, passwords, tokens, or other credentials in the URL;
- keep command-provider transport disabled unless explicitly enabled;
- require TLS for non-local HTTP JSON providers;
- label cloud STT/TTS providers clearly before enabling them;
- treat synthesized audio and TTS cache entries as private household data;
- exclude raw synthesized text from logs by default.

## Widget Security

Jute's widgets are compiled directly into the display application within Jute's unified monorepo widgets library. They do not run in isolated iframes, which makes it critical that:

- Widgets only access approved home data via Jute's server APIs.
- Widgets do not access raw secrets, raw keys, or keys in local storage.
- Widgets request and declare permissions in their Go/Svelte codebase.
- Widgets declare their agent-facing capabilities through safe Widget Skills.

## Theme And Background Security

Theme Packs are data-only visual records. Backgrounds are local-first display assets.

Rules:

- do not load JavaScript, HTML, SVG scripts, remote imports, or executable hooks from Theme Packs;
- do not allow theme metadata to become A2A or MCP instructions;
- do not store raw credentials, household secrets, or agent prompt text in Theme Pack metadata;
- reject remote background image URLs for v1;
- keep file-based backgrounds inside a hub-managed backgrounds directory;
- validate local background paths before exposing them to the display;
- treat user-provided background images as private household media.

## Agent Context Redaction

The hub builds A2A dashboard context from safe, declared data only.

Never send:

- raw credentials;
- hidden widget state;
- private widget fields;
- raw smart-home payloads that include sensitive metadata;
- camera frames;
- microphone audio;
- voice pre-roll buffers;
- partial transcripts;
- synthesized audio cache entries;
- TTS provider credentials;
- precise presence, location, or identity data unless a future explicit permission allows it;
- browser local storage;
- debug logs.

Send only:

- visible widget IDs and titles;
- layout and focus information;
- locale, timezone, and device profile;
- public widget fields declared in [Widget Skills](widget-skills.md);
- coarse display interaction mode.

The same redaction rules apply to the optional [MCP Bridge](mcp-bridge.md). MCP resources, tools, and prompts must be generated by the hub from safe [Widget Skills](widget-skills.md), not from raw widget manifests or direct display component access.

## MCP Bridge Security

The Jute MCP Bridge is disabled by default and local-first.

Rules:

- bind to `127.0.0.1` by default;
- require explicit config before enabling;
- require token auth by default;
- store MCP tokens as secret references only;
- do not use token passthrough;
- do not expose remote cloud agents to local MCP automatically;
- require explicit config and auth for LAN exposure;
- do not expose MCP on the public internet in v1;
- do not allow MCP clients to access raw SQLite data, raw adapter payloads, display DOM, microphone audio, camera frames, or browser storage;
- keep MCP tool descriptions hub-authored to reduce tool-poisoning risk;
- expose widget operations through validated Widget Skills instead of direct display DOM access.

## Agent Trust

Agents are external systems. Jute should present agent trust information clearly:

- source URL;
- selected protocol binding;
- declared skills;
- declared extensions;
- auth configured status;
- last Agent Card refresh;
- last successful health check.

An agent that requires an unsupported extension, unsupported binding, or missing credentials is visible but unavailable.

## Logging

Logs should support debugging without leaking household data.

Default logs include:

- service start and stop;
- config validation errors;
- agent health changes;
- widget install and permission changes;
- task lifecycle IDs.

Default logs exclude:

- raw prompts unless explicit debug logging is enabled;
- secrets and tokens;
- full widget context payloads;
- raw home adapter payloads;
- media content.

## User-Facing Error Redaction

Error UX is public display output. The display must not show raw internal errors, stack traces, SQL errors, token names, credential references, full remote URLs with query strings, raw Agent Cards, widget private state, raw adapter payloads, or raw prompt/transcript/context payloads.

The hub should map internal failures to stable issue codes and safe public messages. The display renders only those safe fields. Full diagnostics belong in future explicit debug surfaces, not dashboard banners, chat rows, or widget frames.

Runtime failure behavior is specified in [Resilience And Error UX](resilience-error-ux.md).

## Backups And Export

Exports should separate:

- portable config and layouts;
- widget installation records;
- SQLite runtime data;
- secrets, which are excluded by default.

Encrypted export can be added later when user profiles and remote sync exist.

YAML/JSON import-export details are specified in [Configuration And Persistence](configuration-persistence.md).
